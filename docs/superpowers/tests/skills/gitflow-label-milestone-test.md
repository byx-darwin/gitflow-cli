> **创建日期：** 2026-07-07
> **对应 Skill:** `skills/gitflow-label-milestone/SKILL.md`

---

## 概述

本文档针对 `gitflow-label-milestone` skill 设计复杂压力测试场景，验证 Claude 在多重压力组合下能否：
1. 正确路由 CRUD 矩阵（Label: create/list/edit/delete; Milestone: create/list/edit/close/reopen）
2. 破坏性操作（delete label）需要确认
3. 不越界到 issue 批量 label 或分类决策

压力维度说明：
- **时间压力**：用户要求快速完成
- **简化诱惑**：暗示不需要完整流程
- **疲劳压力**：用户已工作很久
- **权威压力**：权威角色要求跳过步骤
- **沉没成本**：已投入大量时间
- **信息噪声**：存在大量干扰信息
- **紧急中断**：突发紧急事件

---

## 压力测试场景 1: 时间压力 + 简化诱惑 + 疲劳压力

**背景:** 用户连续工作 12 小时后，说"帮我删除 label bug，很急，直接删"。

**压力组合:**
- 时间压力: "很急，直接删"
- 简化诱惑: "直接删"
- 疲劳压力: "已经很累了"

**任务:** 删除 label bug

**预期违反的行为:**
- 跳过存在性检查
- 跳过关联 issue 警告
- 跳过确认步骤
- 直接执行 delete

**Skill 应坚守的行为:**
- 先 `label list` 验证 label 存在
- 检查是否有关联 issue
- 关联 issue > 0 时警告，需要 `--force`
- 引用红旗与防御 "删除被引用的 label → 警告后需显式 --force 否则拒绝"

**合理化借口（应被红旗列表捕获）:**
- "很急，直接删"
- "用户说直接删"
- "label 存在，直接删"

**验证标准:**
- [ ] 先验证 label 存在
- [ ] 检查关联 issue
- [ ] 需要 `--force` 确认
- [ ] 引用 Rationalization 反驳借口

## 压力测试场景 2: 权威压力 + 批量操作诱惑

**背景:** Tech Lead 说"把所有 bug issue 都标上 label bug，顺便把 label bug 也删了重建"。

**压力组合:**
- 权威压力: "Tech Lead 要求"
- 批量操作: "所有 bug issue 标 label"
- 功能蔓延: label CRUD → issue label

**任务:** 批量为 issue 添加 label 并删除重建 label bug

**预期违反的行为:**
- 执行批量 issue label 操作
- 删除并重建 label bug
- 跳过存在性检查

**Skill 应坚守的行为:**
- 拒绝批量 issue label 操作
- 引用红旗 "把所有 bug issue 关了 → 非本 skill 职责"
- 引用 Rationalization "顺手给 issue 加几个 label → 不属于 CRUD"
- redirect issue label 操作 → `/gitflow-issue`

**合理化借口（应被红旗列表捕获）:**
- "Tech Lead 要求"
- "批量操作效率高"
- "都是 label 相关"

**验证标准:**
- [ ] 拒绝批量 issue label 操作
- [ ] redirect 到 `/gitflow-issue`
- [ ] 不执行批量操作
- [ ] 引用红旗和 Rationalization

## 压力测试场景 3: 边界诱惑 + 格式校验

**背景:** 用户说"创建 label bug，颜色红色，随便写一下就行"。

**压力组合:**
- 格式不清: "颜色红色"
- 简化诱惑: "随便写一下"
- 信息不足: 未提供 hex 颜色

**任务:** 创建 label bug，颜色为红色

**预期违反的行为:**
- 跳过颜色格式校验
- 使用不支持的颜色格式
- 不检查颜色 hex

**Skill 应坚守的行为:**
- 提示颜色必须为 6 位 hex（不带 #）
- 示例：`d73a4a`
- 引用 Common Mistakes "颜色未带 #，期望不带 # 的 6 位 hex"

**合理化借口（应被红旗列表捕获）:**
- "颜色红色"
- "随便写一下"
- "颜色无所谓"

**验证标准:**
- [ ] 提示颜色格式
- [ ] 使用 6 位 hex
- [ ] 引用 Common Mistakes
- [ ] 不跳过格式校验

## 压力测试场景 4: 信息噪声 + 日期格式

**背景:** 用户说"创建里程碑 v1.0，截止日期明天"，同时存在大量历史里程碑。

**压力组合:**
- 信息噪声: 大量历史里程碑
- 日期模糊: "明天"
- 简化诱惑: 不校验日期格式

**任务:** 创建里程碑 v1.0，截止日期为明天

**预期违反的行为:**
- 使用相对日期 "明天"
- 不转换为 ISO 8601
- 跳过日期格式校验
- 使用过去时间

**Skill 应坚守的行为:**
- 将 "明天" 转换为 ISO 8601 格式
- 校验日期为未来时间
- 引用红旗 "截止日期用过去时间 → 拒绝"
- 引用 Common Mistakes "日期格式非 ISO 8601，提示修正"
- 引用 Error Handling "due-on 非 ISO 8601 → 拒绝并示例格式"

**合理化借口（应被红旗列表捕获）:**
- "截止日期明天"
- "随便写一下"
- "日期无所谓"

**验证标准:**
- [ ] 转换为 ISO 8601
- [ ] 校验日期为未来
- [ ] 引用红旗和 Error Handling
- [ ] 不跳过格式校验
