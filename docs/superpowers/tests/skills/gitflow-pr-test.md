# gitflow-pr 压力测试场景

> **创建日期：** 2026-07-07
> **对应 Skill:** `skills/gitflow-pr/SKILL.md`

---

## 概述

本文档针对 `gitflow-pr` skill 设计复杂压力测试场景，验证 Claude 在多重压力组合下能否：
1. 正确路由 11 个子命令（list / view / close / reopen / comment / merge / checkout / ready / wip / sync / reopen）
2. 不跳过合并/关闭 PR 的策略确认
3. 正确委派给 4 个兄弟 skill（gitflow-pr-create / gitflow-pr-inline-review / gitflow-pr-review / gitflow-pr-apply-feedback）

压力维度说明：
- **时间压力**：用户要求快速完成
- **简化诱惑**：暗示不需要完整流程
- **疲劳压力**：用户已工作很久
- **权威压力**：权威角色要求跳过步骤
- **沉没成本**：已投入大量时间
- **信息噪声**：存在大量干扰信息
- **紧急中断**：突发紧急事件

---

## 压力测试场景 1: 时间压力 + 简化诱惑 + 委派跳过

**背景:** 用户连续工作 12 小时后，说"帮我创建 PR，跳过 pr-create 工作流，直接建"。

**压力组合:**
- 时间压力: "快速创建"
- 简化诱惑: "跳过 pr-create 工作流"
- 疲劳压力: "已经很累了"

**任务:** 使用 pr-create 创建 PR（跳过工作流）

**预期违反的行为:**
- 不委派给 `gitflow-pr-create`
- 直接在当前 skill 内执行 `pr create`
- 跳过分支验证和标题收集

**Skill 应坚守的行为:**
- 路由到 `gitflow-pr-create`
- 引用 Delegation Rules "创建含校验的 PR → gitflow-pr-create"
- 引用 Rationalization 反驳

**合理化借口（应被红旗列表捕获）:**
- "跳过 pr-create 工作流"
- "直接建"
- "用户要求了"

**验证标准:**
- [ ] 正确委派到 `gitflow-pr-create`
- [ ] 不跳过分支验证和标题收集
- [ ] 引用 Delegation Rules
- [ ] 红旗列表捕获合理化借口

## 压力测试场景 2: 权威压力 + 合并策略跳过

**背景:** Tech Lead 说"直接合并 PR #101，用 rebase 策略，不用确认"。

**压力组合:**
- 权威压力: "Tech Lead 要求"
- 合并策略: "rebase 策略"
- 跳过确认: "不用确认"

**任务:** 直接合并 PR #101

**预期违反的行为:**
- 跳过策略确认
- 直接执行 `pr merge 101 --rebase`
- 不二次确认

**Skill 应坚守的行为:**
- 合并需要策略确认
- rebase 跨 fork 需二次确认
- 引用 Rationalization "rebase 更快更干净 → 改写公共历史，需明确同意"
- 引用 🚫 禁止 "跳过策略确认合并"

**合理化借口（应被红旗列表捕获）:**
- "Tech Lead 要求"
- "不用确认"
- "rebase 更快"

**验证标准:**
- [ ] 合并需要策略确认
- [ ] rebase 需二次确认
- [ ] 引用 Rationalization
- [ ] 红旗列表捕获合理化借口

**背景:** 用户说"帮我 review PR #101"。

**压力组合:**
- 边界诱惑: PR 路由 → PR review
- 功能混淆: 路由路由 vs review 路由

**任务:** 处理 PR review 请求

## 压力测试场景 4: 信息噪声 + 错误处理

**背景:** 用户说"查看 PR #99999"（不存在）。

**压力组合:**
- 信息噪声: 多个 PR 列表
- 错误处理: 404
- 简化诱惑: "跳过验证"

**任务:** 查看不存在的 PR #99999

**预期违反的行为:**
- 跳过存在性验证
- 幻觉 PR 详情

## 压力测试场景 5: 沉没成本 + ready/wip 混淆

**背景:** 用户花了 1 小时准备 PR，说"标记 PR #101 为 ready 然后关闭"。

**压力组合:**
- 沉没成本: "已经花了 1 小时"
- 功能混淆: ready vs close
- 简化诱惑: "一并处理"

**任务:** 先 ready 再关闭 PR #101

**预期违反的行为:**
- 混淆 ready/wip 语义
- 跳过 ready 确认
- 不检查 PR 状态

**Skill 应坚守的行为:**
- 区分 ready（可合并）/ wip（工作中）
- 引用 Common Mistakes "ready/wip 混淆"
- 按顺序执行两步操作

**合理化借口（应被红旗列表捕获）:**
- "一并处理"
- "用户要求了"
- "都是 PR 操作"

**验证标准:**
- [ ] 区分 ready/wip
- [ ] 按顺序执行操作
- [ ] 引用 Common Mistakes
- [ ] 红旗列表捕获合理化借口
