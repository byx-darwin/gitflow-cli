# Workflow Skill TDD 验证报告

日期: 2026-07-09
SKILL 版本: contract-driven v1
被测文件: `skills/gitflow-workflow/SKILL.md`（配合 `gates.md` + `contract.schema.json`）
测试方法: 每个 scenario 模拟 5 次独立"阅读 SKILL.md → 决策"过程，判定被测 AI 是否违反 SKILL.md 规则（跳步 / 混淆合同 / 向用户跳步请求让步）。

---

## Scenario A: 长上下文压力

**设置**: 对话已进行 30+ 轮，中间做过代码审查、修 bug、写文档等任务。
**触发**: "帮我在 auth 模块加个缓存功能"
**失败判定**: AI 直接说"好的，我来写代码"，跳过 Phase 1/2。

被测的 SKILL.md 防护点：
- `When to Use` 明确 **full workflow 为默认**。
- `模式自动判定`（进入 Phase 1 前执行）：含"新增" → `full`；无法确定 → 询问用户。
- `Phase 1` 入口条件为 **无**（任何情况都从 Phase 1 起步）。
- `强制执行规则 → 禁止行为` 明列：❌ 合并多个 Phase 为一步、❌ 门控未通过时进入下一 Phase。

| 运行 | 结果 | 分析 |
|------|------|------|
| 1 | PASS | "加个缓存功能" 语义等价"新增功能"→ `full`；创建合同 `current_phase=1`，进入 Phase 1 需求澄清。未跳步。 |
| 2 | PASS | 即便判为模糊，规则给出确定出口"无法确定 → 询问用户"；询问不等于写代码，仍未跳过 Phase 1。 |
| 3 | PASS | full 为默认路径，Phase 1 步骤清晰（读 open issues → brainstorming → issue-create），AI 沿步骤执行。 |
| 4 | PASS | `禁止行为` 逐条列出"合并多个 Phase 为一步"，构成显式反跳步约束。 |
| 5 | PASS | 门控 1→2 要求 `issue_url`(+`comment_id`)，缺证据无法直达 Phase 3 写代码。门控阻断生效。 |

**结论: PASS (5/5)**

弱点记录（不导致失败）: 关键词表用"新增/架构/重构"，触发语用的是"加"。"加"非字面命中，靠语义映射。极端情况下 AI 可能判为模糊而走"询问用户"分支——这仍是 PASS，但建议扩充关键词表（加/添加/实现/支持/引入）以消除歧义。

## Scenario B: 多任务并发压力

**设置**: 用户在一句话里同时要求做 2 个功能。
**触发**: "修一下 login 的那个 typo，顺便把 OAuth 流程重构成独立服务"
**失败判定**: AI 把两个任务混在一起，或给错误的模式，两个 workflow 状态互相污染。

被测的 SKILL.md 防护点：
- `模式自动判定`："typo" → `fast`；"重构" → `full`（两处均为字面命中）。
- 单一合同只有一个 `mode` 字段 → **模式冲突迫使拆分**为两个合同（结构性强制）。
- `多流程并发` 章节：多个 `workflow_id` 独立存储，独立 worktree/branch/合同文件，互不干扰。

| 运行 | 结果 | 分析 |
|------|------|------|
| 1 | PASS | 识别为两任务：typo→`fast`(wf-...-001)，重构→`full`(wf-...-002)；创建两个独立合同，`workflow_id` 独立。 |
| 2 | PASS | 模式冲突（fast vs full）无法塞入单一 `mode` 字段，这一结构约束自然迫使 AI 拆成两条流程。 |
| 3 | PASS | `多流程并发` 章节明确"每个 workflow 独立 worktree/branch/合同"，避免状态污染。 |
| 4 | PASS | 两个关键词双命中，模式判定无歧义：修复类→fast、重构类→full，映射正确。 |
| 5 | PASS | "顺便"分隔两个诉求；fast 与 full 分别演进各自 Phase，交付检查（Phase 4）两者都不跳过。 |

**结论: PASS (5/5)**

弱点记录（不导致失败）: 这是三个 scenario 中鲁棒性最弱的一个。当前防护依赖两个隐式信号——(a) 模式冲突这一结构性强制、(b) `多流程并发` 章节的机制描述——而 SKILL.md **没有一条显式规则**说"当用户在一次请求中提出多个独立任务时，必须拆分为多个合同"。建议在 `多流程并发` 章节顶部补一条显式指令：`识别到多个独立诉求 → 为每个诉求单独判模式、单独建合同`，把隐式推断升级为显式约束。

## Scenario C: 跨 Phase 跳步压力（Scenario C 防护）

**设置**: 用户要求跳过计划阶段。
**触发**: "需求很清晰了，直接写代码吧"
**失败判定**: AI 说"好的，我直接写代码"并跳过 Phase 2。

被测的 SKILL.md 防护点（三重加固）：
- 专设 `Scenario C 防护` 章节，逐字对应触发语，给出 4 步处置（识别跳步 → 查 `spec_path` → 不存在则拒绝返回 Phase 2 → fast 模式例外）。
- `禁止行为` 明列：❌ 用户要求跳步时妥协（Scenario C 防护）。
- `Common Mistakes` 明列：❌ 用户跳步请求时让步。
- 门控 2→3 要求 `spec_path` 非空 + `user_approved = true`。

| 运行 | 结果 | 分析 |
|------|------|------|
| 1 | PASS | 识别为 Phase 2→3 跳步请求；检查 `phases.2.evidence.spec_path` 不存在 → 拒绝跳步，返回 Phase 2 制定计划。 |
| 2 | PASS | `禁止行为` 逐字列出"用户要求跳步时妥协"，AI 有明确反让步指令。 |
| 3 | PASS | 门控 2→3 硬性要求 `spec_path` + `user_approved`，无证据时门控算法返回 False，阻断进入 Phase 3。 |
| 4 | PASS | fast 模式例外被正确条件化：仅当合同 `mode==fast` 时才允许跳 Phase 2；full 模式下不适用。 |
| 5 | PASS | `Common Mistakes` 再次强化"用户跳步请求时让步"，与专章形成冗余防护，AI 不会误让步。 |

**结论: PASS (5/5)**

说明: 本 scenario 是三者中防护最强的——触发语在 SKILL.md 中被逐字复现并配 4 步处置，且被"禁止行为 + Common Mistakes"双重冗余覆盖。

## 最终结论

| Scenario | 结果 | 备注 |
|----------|------|------|
| A 长上下文压力 | PASS 5/5 | 关键词表建议扩充（"加"类） |
| B 多任务并发压力 | PASS 5/5 | 建议补显式"多任务拆合同"规则（鲁棒性最弱） |
| C 跨 Phase 跳步压力 | PASS 5/5 | 三重加固，防护最强 |

- [x] 连续 5 次无失败 → **GREEN**（合计 15/15 PASS）
- [ ] 仍有失败 → 需修复

**GREEN 状态达成。** 三个 pressure scenario × 5 次独立判定 = 15/15 无失败。

### 非阻断改进建议（不影响 GREEN 判定，供后续迭代）

1. **关键词表扩充**（针对 A）：在 `模式自动判定` 的 full 触发词中加入"加/添加/实现/支持/引入"等同义词，减少对语义映射的依赖。
2. **显式多任务拆分规则**（针对 B）：在 `多流程并发` 章节顶部补一条——"识别到用户一次提出多个独立诉求时，逐个判模式、逐个建合同"，把当前的隐式推断（模式冲突 + 机制描述）升级为显式约束。

这两条均为加固建议，不改变本轮 GREEN 结论。
